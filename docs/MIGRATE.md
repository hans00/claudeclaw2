# Migrate v1 → v2

If you already run v1 ClaudeClaw and want to keep your existing
conversations and per-channel sessions, this is your path. The
auto-migration runs on the daemon's first boot.

## What gets carried over

| v1 | v2 |
|----|----|
| `.claude/claudeclaw/session.json` (global) | `sessions["global"]` |
| `.claude/claudeclaw/sessions.json.threads.<id>` (Discord per-thread) | `sessions["discord:<id>"]` (multiparty=true) |
| `.claude/claudeclaw/settings.json` | same path; v2 reads compatible subset |
| `.claude/claudeclaw/inbox/*.jsonl` | same path; v2 cross-session inbox is identical |
| `.claude/claudeclaw/jobs/*.md` | same path; same frontmatter format |
| Claude Code session jsonl in `~/.claude/projects/<cwd>/<uuid>.jsonl` | reused directly — sessions resume with full context intact |

## What is NOT carried over

- v1 daemon state (turn count, compactWarned) — these were per-session
  bookkeeping; v2 starts fresh and lets Claude Code's own compact
  warning handle it
- v1 `state.json` — regenerated by v2's statusline writer
- v1 `daemon.pid` — replaced on first v2 start
- whisper transcripts — v2 hands voice files to the agent as-is; see
  [TODO.md](TODO.md)
- LINE config — v1 didn't have LINE saved in settings; if you used it,
  re-enter the tokens

## Steps

### 1. Stop v1

```bash
cd /path/to/your/project    # the directory with .claude/claudeclaw/
kill -TERM $(cat .claude/claudeclaw/daemon.pid)
```

Or whatever stop method you use today.

### 2. Get the v2 source

```bash
git clone https://github.com/hans00/claudeclaw2.git ~/Projects/claudeclaw2
cd ~/Projects/claudeclaw2
bun install
```

### 3. Point your start.sh at v2

Edit your existing `start.sh` (or run `init` to generate a fresh one in
a different directory — see [INSTALL.md](INSTALL.md)). The minimal v2
launcher is:

```bash
#!/bin/bash
cd "$(dirname "$0")"
mkdir -p .claude/claudeclaw/logs

nohup bun run ~/Projects/claudeclaw2/src/daemon.ts \
  > .claude/claudeclaw/logs/daemon.log 2>&1 &
```

Note: v2 has no `start --web` flag. The web dashboard is enabled by
`web.enabled: true` in `settings.json` (default port `4632`). If your v1
settings file is missing the `web` block, add it.

### 4. First-boot auto-migration

```bash
./start.sh
tail -f .claude/claudeclaw/logs/daemon.log
```

On the first boot you should see something like:

```
[migrate] copied v1 global session → .claude/claudeclaw/session.json.v1-backup
[migrate] backed up v1 threads file → .claude/claudeclaw/sessions.json.v1-backup
[migrate] migrated 2 entries:
  + global → 37c1ade4
  + discord:1492947673311871069 → 256fbd5f
[daemon] restoring 2 session(s)...
[daemon] restored global → claudeclaw-global
[daemon] restored discord:1492947673311871069 → claudeclaw-discord-1492947673311871069
```

The originals are renamed `*.v1-backup` so you can roll back. Subsequent
restarts will log `[migrate] skipping: v2 sessions.json already
populated`.

### 5. Verify

```bash
# both sessions should be live
tmux ls

# the dashboard should list them
curl -s http://127.0.0.1:4632/api/sessions | python3 -m json.tool

# send a Telegram message and watch it route to global
```

If a session was huge (long-running v1 global with hundreds of turns),
the first `claude --resume <uuid>` may take 10–20 seconds to come up
while it loads the jsonl. The init-watch waits and may print
`[init-watch] pane stuck for 15000ms` and send an Esc; that's harmless
unless the session never reaches ready state.

### 6. Roll back if needed

```bash
# stop v2
bun run ~/Projects/claudeclaw2/src/cli.ts stop

# restore v1 files
mv .claude/claudeclaw/session.json.v1-backup .claude/claudeclaw/session.json
mv .claude/claudeclaw/sessions.json.v1-backup .claude/claudeclaw/sessions.json
rm -f .claude/claudeclaw/sessions.json   # the v2 one
# (rename your v1 sessions.json back if you renamed it)

# restart v1 with the v1 start.sh
./start.sh
```

The Claude Code session jsonl files in `~/.claude/projects/<cwd>/`
are untouched throughout, so v1 sees exactly what it had before.

## Things to know

**v1 and v2 use the same `.claude/claudeclaw/settings.json` path.** v2
reads a superset of the fields v1 had — extra keys (`agentic`, `model`,
`fallback`, `heartbeat.timezone`) are preserved, unknown keys are
ignored. You don't need to rewrite the file.

**v1 features not in v2.** See [TODO.md](TODO.md). The notable ones:
- whisper voice transcription (v2 hands the audio file path to the agent)
- v1's full `proxy` command
- web UI history viewer (v2 has a simpler dashboard at `/sessions/:key`)
- agent-callable MCP tools for cron (v2 lets the agent write the .md
  job file directly using Write/Bash — same effect, less plumbing)

**v2 routing parity check.** v1 sent all Telegram + Discord DMs to the
global session. v2 does the same. If you had a Discord channel-level
session in v1 (it had its own jsonl), v2 preserves that as
`discord:<channelId>`. Slack threads in v2 get their own session
(`slack:<channelId>:<threadTs>`) — this matches v1 thread behavior.
